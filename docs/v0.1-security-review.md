# v0.1 security review

- **Date:** 2026-05-09
- **Reviewer:** mt-commerce maintainers (assisted by automated audit)
- **Scope:** `apps/api`, `apps/admin`, `apps/storefront`, `packages/sdk`, `packages/core`, `packages/plugins/*`

This is the pre-launch paper review of mt-commerce v0.1. It records what was checked, what was found, and what is verified to be in good shape.

The review is structured around the items called out in the v0.1 checklist: secrets, rate limits, idempotency, webhook signatures, and input validation — extended with the rest of the OWASP Top 10 surface that applies to a commerce API of this shape.

The review does not replace a paid third-party audit. It is the engineering team's structured pass on the codebase as it stands at the v0.1 launch candidate.

## 1. The IDOR on `/customer/me/*` (HIGH)

### What it is

Three storefront-side modules — `customer`, `cart`, `orders` — resolve "the current customer" by reading the `x-customer-id` HTTP header. The header is client-controlled. An attacker holding any other customer's ULID can:

- read the victim's profile, addresses, cart, and orders;
- modify the victim's addresses (PATCH/DELETE);
- add items to or remove items from the victim's cart.

This is a classic insecure direct object reference. Customer ULIDs are not secret — they are returned in API responses and stored client-side; an attacker who has placed a single test order knows the format and can guess or harvest IDs.

The header was introduced as a v0.1 stand-in until Better Auth's customer-side session flow was wired through. The auth scaffolding (`requireAuth()`, `getAuthedUser`, `getCustomerByAuthUserId`) exists in `apps/api/src/modules/auth/`; the storefront SDK already sends `credentials: "include"`. The deferred work is the migration: replace `resolveCurrentCustomerId` with auth-derived lookup in the three modules, drop `x-customer-id` from the SDK, and verify session cookies traverse the storefront → API call path.

### Where it lives

- `apps/api/src/modules/customer/routes/storefront.ts:65-81` — `resolveCurrentCustomerId` function and every call site.
- `apps/api/src/modules/cart/routes/storefront.ts:65-83` — same pattern.
- `apps/api/src/modules/orders/routes/storefront.ts:38` — `CUSTOMER_HEADER` constant; same pattern.
- `packages/sdk/src/client.ts:1956+` — SDK forwarder that injects the header.
- `apps/storefront/src/lib/account.ts:27` — localStorage cache that supplies it.

Three module READMEs (`customer/README.md:132`, `cart/README.md:126`, `orders/README.md:124`) call the issue out as a TODO with the planned fix.

### Required for launch

This is a launch blocker. The customer-auth integration must land before v0.1 is exposed to production traffic. The fix is feature-track work, not a security patch — it requires:

1. Update the three route files to use `requireAuth()` + `getCustomerByAuthUserId(authUser.id)`.
2. Drop the `x-customer-id` forwarder in the SDK; rely on cookie credentials.
3. Update the storefront sign-in flow to verify the session cookie is delivered to the API on subsequent calls (it should, via `credentials: "include"`).
4. Migrate all tests that pass `x-customer-id` to use a Better Auth session fixture.

The work is sized as a single coherent track and tracked separately. This document records that v0.1 cannot ship without it.

## 2. In-memory rate-limit store (LOW)

The global rate-limit middleware (`apps/api/src/middleware/rate-limit.ts:17-18`) keeps its bucket state in process memory. This is correct for development and single-process deployments. It does not survive a process restart and is not shared between processes.

For v0.1's single-process VPS deployment target, this is acceptable. For any multi-process deployment (or a deployment behind a load balancer that load-balances at the request level), the buckets are per-process and trivial to circumvent — split the attack across N processes, get N× the throughput.

The fix is to back the limiter with Redis (which the deployment already runs for sessions). The TODO is in the middleware comment.

**Required action:** Before any multi-process or horizontally-scaled deployment, swap the in-memory store for the Redis-backed equivalent. v0.1's single-process deployment can ship without this fix; the deployment guide should call it out as a prerequisite for scaling.

## 3. Image upload idempotency (INFO)

The product image upload route (`POST /admin/v1/products/{id}/image`) does not mount `requireIdempotencyKey`. The audit flagged this as a missing protection.

On review, the design already protects retries:

- The filename is `sha256(bytes).slice(0, 16) + ext`. Two uploads of the same bytes write to the same path — the disk dedupes.
- The product row's `imageUrl` update is set-once (last write wins). Two retries of the same upload set the same URL.
- Two retries of _different_ bytes are not a retry; they are two distinct uploads, and the operator's intent is "use the second one." Idempotency does not apply.

Adding `Idempotency-Key` would force the admin client to mint a key for every upload click. The friction exceeds the value because the design is already retry-safe. **No action.**

## What was verified

The remaining categories were checked and are in good shape. Each line below is a one-liner; specifics live in the module they reference.

### Secrets

- `.env.example` files carry only placeholder values — `BETTER_AUTH_SECRET=change-me-to-32-chars`, no real tokens.
- Env validation runs via Zod at startup (`apps/api/src/lib/env.ts`); missing required vars fail boot loudly.
- No secrets are logged at info level. Pino redaction is configured for the auth module's verification URLs (logged at debug only).
- `git log -p -- .env.example` shows no historical leaks.

### Rate limiting

- Better Auth owns `/api/auth/*` with tight per-IP rules (sign-in 5/60s, password-reset 3/300s).
- Global limiter covers all other routes (120/60s default).
- Pre-flight `OPTIONS` short-circuits before rate checks (CORS preflight does not eat rate-limit budget).
- The webhook receivers do not impose route-level rate limits — provider redeliveries can spike past the limit, and the design relies on signature verification + `(providerRef, status)` dedup at the data layer to absorb the burst safely.

### Idempotency-Key middleware (where it matters)

Mounted on every state-mutating endpoint with a non-trivial side effect:

- `POST /storefront/v1/checkout/{id}/complete` (`scope: "checkout.complete"`)
- `POST /admin/v1/payments/{id}/capture` (`scope: "payment.capture"`)
- `POST /admin/v1/payments/{id}/refund` (`scope: "payment.refund"`)
- Storefront `payments.initiate` carries the row-level `payments.idempotency_key` column as the durable backstop; the admin actions ride the HTTP middleware.

Webhook receivers explicitly do not use the middleware — providers don't supply an `Idempotency-Key`. They dedupe at the data layer on `(providerRef, status)` per ADR-0012.

ADR-0019 records the dual-layer design.

### Webhook signature verification

Every webhook receiver verifies its provider's signature **before** parsing the body or trusting any field:

- **Midtrans** — SHA-512 of `order_id + status_code + gross_amount + serverKey`, constant-time comparison via `crypto.timingSafeEqual`. (`packages/plugins/payment-midtrans/src/signature.ts`.)
- **Biteship** — HMAC-SHA256 with `timingSafeEqual`. Signature header lookup is case-insensitive.
- **In-memory test provider** — also uses `timingSafeEqual` for parity.

No `===` comparisons on signatures anywhere in the codebase. No webhook handler reaches into the body before signature verification has succeeded.

### Input validation

- Every request body, query string, and route param is validated through `@hono/zod-openapi` with a Zod schema. The OpenAPI document is generated from the same schemas.
- No `c.req.json()` calls that bypass validation.
- No `as` casts on incoming data that defeat the validator.
- Drizzle's parameterized queries prevent SQL injection by construction; raw `sql` template strings (used sparingly) carry only column/table identifiers, never user input.

### Auth & access control

- Every `/admin/v1/*` route is gated by `requireAuth()` + `requireRole(...)`.
- API keys with explicit scopes are an alternative path; `requireScope(scope)` middleware checks the scope set.
- Order/refund/capture endpoints stamp the audit log with `actorKind` + `actorId` from the auth context (not from the request body).
- The IDOR on `/customer/me/*` (above) is the exception, tracked separately.

### CORS

- `CORS_ORIGIN` env var is the source of truth. Production rejects wildcard `*` when credentials are enabled (the middleware throws at boot).
- Dev defaults to localhost-only origins.

### Cryptographic correctness

- All HMAC comparisons use `crypto.timingSafeEqual` on `Buffer` representations (constant-time, length-checked).
- Argon2id is the default through Better Auth — framework defaults are current and adequate.
- `crypto.randomUUID()` is used for idempotency keys; `id("…")` for ULIDs.

### Logging & monitoring

- The standard error envelope (`{ error: { code, message, details } }`) sanitizes unhandled exceptions to a generic `internal_error`. Stack traces never reach the client.
- Pino structured logging carries `requestId`, `userId`, `module` — no passwords, tokens, or signatures.
- The audit log records every staff mutation with the actor stamp from the auth context.

### CSRF

- Better Auth sets session cookies as `HttpOnly; Secure; SameSite=Lax`. The lax mode allows top-level navigations to authenticated pages while blocking cross-site form posts.
- API mutating endpoints accept session cookies only same-origin (CORS gate).

### File upload

- Size capped via `MAX_UPLOAD_BYTES` (default 5 MiB), refused before bytes are read.
- Accepted MIME types: `image/jpeg`, `image/png`, `image/webp`. **SVG is excluded by design** (it can carry script).
- Magic-byte validation runs against the declared MIME — a renamed `.html` with `Content-Type: image/png` is rejected.
- Filenames are content-hashed (SHA-256 prefix); they are not derived from user input.

### SSRF

- No outbound `fetch()` calls take user-supplied URLs as input.
- The product editor's `imageUrl` is a Zod-validated `http(s)` URL, but the API does not fetch it — the storefront renders it directly. (When the user uploads instead of pasting a URL, the upload pipeline is the ingress, and the URL it stores is fully constructed by the API.)

### Storefront / admin client surface

- No `dangerouslySetInnerHTML`, `innerHTML =`, `eval`, or `new Function` in either app's React source.
- No tokens stored in `localStorage`. Theme + locale + cached customer ULID (the IDOR finding above) live there; auth tokens are HttpOnly cookies.

## Out of scope for this review

- **Penetration testing.** No active exploit attempts were run. The review is paper + grep + targeted code reading, not a network-level test.
- **Dependency vulnerability scan.** A separate `bun audit` / Dependabot pass is the right channel for this. None of the audited code introduces a vulnerable pattern that a dependency-version audit would have caught instead.
- **Container / image hardening.** The Dockerfile review is a deployment-side concern.
- **Legal & compliance.** PDPL/GDPR coverage of the data model is a separate workstream.

## Sign-off conditions for v0.1

The audit lists one launch blocker and one tracked follow-up:

1. **Customer-auth integration must land** before v0.1 is exposed to public traffic. The IDOR on `/customer/me/*` is exploitable today.
2. **Redis-backed rate limiter must land** before any multi-process deployment.

Once those are in, v0.1 is in adequate security shape for the merchant audience the platform is designed for. A subsequent third-party audit before v1.0 is recommended.
