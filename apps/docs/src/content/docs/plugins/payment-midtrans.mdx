`@mt-commerce/plugin-payment-midtrans` mendaftarkan `PaymentProvider`
dengan code `midtrans` ke loader plugin mt-commerce. Plugin ini
memakai Midtrans Snap (checkout terhosting) untuk alur pembeli dan
Midtrans Core API untuk refund.

## Apa yang dilakukan plugin

- Pada `initiate`, meminta token transaksi Snap dan mengembalikan URL
  redirect Snap yang dipakai storefront untuk mengarahkan pembeli.
- Pada `refund`, mengirim POST ke endpoint Midtrans Core API
  `/v2/{paymentId}/refund`, memakai idempotency key platform sebagai
  `refund_key` Midtrans agar retry menjadi no-op di sisi Midtrans.
- Pada pengiriman webhook, memverifikasi field `signature_key` sesuai
  formula yang didokumentasikan Midtrans
  (`SHA512(order_id + status_code + gross_amount + serverKey)`)
  sebelum platform mem-parse atau mempercayai field apa pun dari body.
- Pada rekonsiliasi, memanggil endpoint Midtrans Core API
  `GET /v2/{orderId}/status` dan mengembalikan `PaymentStatusSnapshot`
  yang diterapkan platform lewat mesin transisi yang sama dengan
  webhook. Dipakai oleh aksi "reconcile" di admin dan job bulk
  `reconcile-pending` untuk memulihkan webhook yang terlewat.
- `capture` adalah no-op terstruktur — Snap auto-capture untuk kanal
  yang aktif secara default.

## Kanal yang aktif default

QRIS, GoPay, ShopeePay, virtual account BCA / BNI / BRI / Permata,
Mandiri Bill, kartu kredit (3DS), Indomaret, Alfamart. DANA mengalir
lewat QRIS; OVO sudah dihapus dari Snap oleh Midtrans. Operator dapat
meng-override set per transaksi via `metadata.enabledPayments`.

## Konfigurasi

```ts
// mt-commerce.config.ts
import { defineConfig } from "@mt-commerce/core/plugin";
import midtrans from "@mt-commerce/plugin-payment-midtrans";

export default defineConfig({
  plugins: [
    midtrans({
      mode: process.env.NODE_ENV === "production" ? "production" : "sandbox",
      serverKey: process.env.MIDTRANS_SERVER_KEY!,
      clientKey: process.env.MIDTRANS_CLIENT_KEY!,
    }),
  ],
});
```

Daftar di [dashboard.midtrans.com](https://dashboard.midtrans.com/register),
lalu ambil kunci sandbox dari **Settings → Access Keys** untuk
pengembangan.

## Merekonsiliasi webhook yang terlewat

Pengiriman webhook dari Midtrans bersifat best-effort. Jika callback
hilang (jaringan terputus, deploy di tengah jalan, perubahan firewall),
baris di platform tetap `pending` sementara Midtrans melihat
`settlement`. Dua permukaan menangani penyelarasan di luar jalur:

- `POST /admin/v1/payments/{id}/reconcile` — merekonsiliasi satu
  payment. UI admin memunculkan tombol ini di halaman detail payment.
- `POST /admin/v1/payments/reconcile-pending` — bulk-poll setiap
  payment yang belum terminal yang `updated_at`-nya lebih lama dari
  `olderThanMinutes` (default 5). Sambungkan ini ke cron host atau
  antrian job yang berjalan setiap beberapa menit.

Kedua endpoint memeriksa status kanonis payment via
`GET /v2/{orderId}/status` dan menerapkan transisi yang sama dengan
webhook. Rekonsiliasi yang menyebabkan perubahan state mencatat baris
`reconcile` di `payment_attempts` sehingga jejak audit membedakannya
dari callback asli.
