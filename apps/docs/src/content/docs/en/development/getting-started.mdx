This page is for contributors — people who will touch the code. If
you only want to try the platform out on a laptop, the
[quickstart](/en/quickstart) is shorter.

## Prerequisites

- [Bun](https://bun.sh/) 1.3.7 or newer. Check with `bun --version`.
  The version this repo pins lives in `.bun-version`.
- [Docker Desktop](https://www.docker.com/) with the Compose plugin.
  Postgres and Redis run as containers during local development.
- Git.

`bun install` handles the rest — no separate Node or npm install is
needed.

## First-time setup

The script that does everything:

```bash
git clone https://github.com/masyarakat-terbuka/mt-commerce.git
cd mt-commerce
bun run setup
```

`bun run setup` is idempotent. What it does, in order:

1. Verifies `bun` and `docker` are on the PATH.
2. Copies `apps/{api,admin,storefront}/.env.example` to `.env` where
   missing (your existing files are kept).
3. Runs `bun install` across the workspace.
4. `docker compose up -d --wait` so Postgres and Redis are healthy.
5. Applies database migrations.
6. Seeds Indonesian regions and a demo catalog.

To start over from a clean slate — including dropping the Docker
volumes — use `bun run reset`.

## Running the stack

```bash
bun run dev
```

This runs the API, admin, and storefront in parallel via Bun's
workspace filter.

| App         | URL                       | Port |
| ----------- | ------------------------- | ---- |
| API         | http://localhost:8000     | 8000 |
| Storefront  | http://localhost:4321     | 4321 |
| Admin       | http://localhost:5173     | 5173 |

For the first admin login, see "Provisioning the first owner" in
[Authentication](/en/api-reference/authentication).

## Workspace layout

mt-commerce is a Bun monorepo. The shape:

```
apps/
  api/         # Hono on Bun — commerce modules behind HTTP
  admin/       # Vite + React SPA
  storefront/  # Astro 5 + React islands
  docs/        # this Starlight site
packages/
  core/        # lightweight cross-module types + the plugin surface
  sdk/         # typed fetch client for the API
  plugins/     # plugin packages (example, midtrans, biteship, whatsapp)
  eslint-config/
  prettier-config/
docs/
  adr/         # Architecture Decision Records
scripts/       # setup, reset, commit-msg hook
```

The full picture lives in [Architecture](/en/architecture).

## Daily commands

Each command in the table below runs across the workspace via the
`bun --filter '*'` filter. Add `--filter @mt-commerce/<package>` to
narrow to one package.

| Command                | Purpose                                |
| ---------------------- | -------------------------------------- |
| `bun run dev`          | Start API + admin + storefront         |
| `bun run build`        | Build every workspace                  |
| `bun run typecheck`    | TypeScript across every package        |
| `bun run lint`         | ESLint                                 |
| `bun run format`       | Prettier (write)                       |
| `bun run format:check` | Prettier (check, for CI)               |

Tests run per-package:

```bash
bun --filter '@mt-commerce/api' test
bun --filter '@mt-commerce/storefront' test
bun --filter '@mt-commerce/plugin-payment-midtrans' test
```

## Conventions

Git hooks enforce the load-bearing conventions automatically. You
rarely need to think about them.

- **Commit messages** follow
  [Conventional Commits](https://www.conventionalcommits.org/). The
  `commit-msg` hook (in `scripts/check-commit-msg.sh`) validates the
  `<type>(<scope>)?: <subject>` shape against a scope whitelist that
  names the workspaces and domains.
- **lint-staged + prettier + eslint** runs in the `pre-commit` hook
  on staged files. Files are formatted and auto-fixed; remaining
  errors block the commit with a clear message.

For the broader PR guidance, see
[`CONTRIBUTING.md`](https://github.com/masyarakat-terbuka/mt-commerce/blob/main/CONTRIBUTING.md)
in the repo.

## Database

Schemas live in `apps/api/src/db/schema/` (one file per table).
Migrations are plain SQL in `apps/api/drizzle/migrations/`,
forward-only, applied by Drizzle's runner. See
[Migration workflow](/en/development/migrations) for the details.

## Plugins

To build a plugin (payment provider, shipping provider, notification
channel, or event listener), see the
[Plugin author guide](/en/plugins/author-guide). The
`@mt-commerce/plugin-example` package in the repo is the smallest
working template.

## When something jams

- `bun run reset` — drops the Docker volumes and re-runs setup. Safe
  any time you don't have data in the local database that matters.
- Port collision? Check for another process on 8000, 4321, or 5173.
  The `dev` script will not silently move ports.
- Healthy database but 500s from the API? Check that `db:migrate` ran
  after your last pull from `main`.
- Hooks failing on a freshly-updated Bun? Delete
  `.git/hooks/pre-commit` and re-run `bun install` —
  `simple-git-hooks` will reinstall it.

## ADRs

Architectural decisions are recorded under `docs/adr/` in the repo.
Each ADR is numbered (e.g. `0010-product-content-translations.md`)
and documents one decision with context, options, and consequences.
A new ADR is added when a decision will affect more people than its
author.
