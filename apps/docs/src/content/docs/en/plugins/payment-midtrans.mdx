`@mt-commerce/plugin-payment-midtrans` registers a `PaymentProvider`
with code `midtrans` against the mt-commerce plugin loader. It uses
Midtrans Snap (hosted checkout) for the buyer flow and the Midtrans
Core API for refunds.

## What the plugin does

- On `initiate`, requests a Snap transaction token and returns the
  Snap redirect URL the storefront sends the buyer to.
- On `refund`, posts to the Midtrans Core API
  `/v2/{paymentId}/refund` endpoint, using the platform's idempotency
  key as Midtrans's `refund_key` so retries are no-ops on Midtrans's
  side.
- On webhook delivery, verifies the `signature_key` field per the
  Midtrans documented formula
  (`SHA512(order_id + status_code + gross_amount + serverKey)`) before
  the platform parses or trusts any field of the body.
- On reconciliation, polls the Midtrans Core API
  `GET /v2/{orderId}/status` endpoint and returns a `PaymentStatusSnapshot`
  the platform applies through the same transition machinery as a
  webhook. Used by the admin "reconcile" action and the bulk
  `reconcile-pending` job to recover from missed webhooks.
- `capture` is a structured no-op — Snap auto-captures for the channels
  enabled by default.

## Channels enabled by default

QRIS, GoPay, ShopeePay, BCA / BNI / BRI / Permata virtual accounts,
Mandiri Bill, credit card (3DS), Indomaret, Alfamart. DANA flows through
QRIS; OVO has been removed from Snap by Midtrans. Operators can
override the set per transaction via `metadata.enabledPayments`.

## Configuration

```ts
// mt-commerce.config.ts
import { defineConfig } from "@mt-commerce/core/plugin";
import midtrans from "@mt-commerce/plugin-payment-midtrans";

export default defineConfig({
  plugins: [
    midtrans({
      mode: process.env.NODE_ENV === "production" ? "production" : "sandbox",
      serverKey: process.env.MIDTRANS_SERVER_KEY!,
      clientKey: process.env.MIDTRANS_CLIENT_KEY!,
    }),
  ],
});
```

Sign up at [dashboard.midtrans.com](https://dashboard.midtrans.com/register),
then grab the sandbox keys from **Settings → Access Keys** for development.

## Reconciling missed webhooks

Webhook delivery from Midtrans is best-effort. If a callback is dropped
(network blip, deploy mid-flight, firewall change), the platform's row
stays `pending` while Midtrans's view says `settlement`. Two surfaces
catch up out of band:

- `POST /admin/v1/payments/{id}/reconcile` — reconcile a single
  payment. The admin UI exposes this from the payment detail page.
- `POST /admin/v1/payments/reconcile-pending` — bulk-poll every
  non-terminal payment whose `updated_at` is older than
  `olderThanMinutes` (default 5). Wire this to a host cron or job
  queue running every few minutes.

Both endpoints look up the payment's canonical state via
`GET /v2/{orderId}/status` and apply the same transitions a webhook
would. A reconciliation that drives a state change records a
`reconcile` row in `payment_attempts` so the audit trail distinguishes
it from a real callback.
